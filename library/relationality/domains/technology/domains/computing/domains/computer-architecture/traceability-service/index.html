Traceability & Knowledge service

Traceability & Knowledge Service Architecture

Maintains and exposes the relational knowledge and traceability state of the Relational Information Physics Computer. This service:

  • presents recognitions, edges, and sheaf sections as a navigable state view;
  • records and exposes dependency relationships between recognitions (tracepoint edges, runbook executions, policy decisions);
  • and enforces policy controls over access and mutation.

Dependencies

  • Audit Simulation: provides immutable logs of recognition/runbook/policy events.
  • Policy Simulation: dictates which roles may read or modify sensitive recognitions and edges.
  • Measurement Fabric Simulation: publishes graph health and query metrics (size, query latency) and mutation telemetry.
  • Topology Integration Simulation: ensures graph exports include the placement/constraint metadata required by downstream topology-aware services.

Components

  1. Graph Builder.
    Reads recognitions and tracepoint_edges from the persistence layer and assembles a typed graph per the traceability-graph specification. Supports filtered views (by acts, domains, policies, topology constraints).

  2. Policy Gate.
    Applies policy-stack rules before exposing or mutating recognitions and edges; coordinates multi-party approvals for privileged operations.

  3. Diff & Justification Engine.
    Shows how proposed changes to recognitions/runbooks/policies affect reachability and invariants, detects missing dependencies, and requires justification before accepting updates.

  4. API Layer.
    Serves MCP/OpenAPI endpoints so other services (runtime, operations control, deployment planner) can:

    • explore the recognition/edge graph;
    • retrieve sheaf-section views and traceability paths for given nodes;
    • and verify that plans and state transitions trace back to earned acts.

Invariants

  • The stored recognition/edge graph is consistent with the traceability-graph specification; missing dependencies are reported as validation errors.
  • Every mutation of recognitions, edges, runbooks, or policies produces an audit event with required approver information and telemetry records.
  • Exposed views are consistent snapshots tagged with topology and measurement metadata so downstream modules can cache, reason about, and verify results.
