YAML Frontmatter as Fregean Predicate Fibration

2026-03-28 by emsenn
Abstract

Markdown files with YAML frontmatter implement Fregean subject-predicate-object triples. Across a vault, predicates index fibers of values, forming a degenerate hyperdoctrine whose emergent structure can be examined without external schema.

<h2 id="abstract">Abstract</h2> <p>A markdown file with YAML frontmatter encodes a set of subject-predicate-object triples: the file is the subject, each key is a predicate, and each value is an object. This is not a metaphor. It is Frege&rsquo;s function-argument structure (Frege 1879) implemented in a plain-text format now standard across millions of repositories.</p> <p>When many files share the same predicate keys, each key generates a <strong>fiber</strong>: the collection of all values that key takes across files. The fiber for <code>authors</code> contains every author who appears in any file&rsquo;s <code>authors</code> field. The fiber for <code>extends</code> contains every concept any file claims to extend. These fibers, indexed by predicate, form a structure that categorical logicians call a fibration (Jacobs 1999) — specifically, a degenerate hyperdoctrine where the base category is the set of predicate keys and each fiber is the set of values assigned to that key.</p> <p>This paper makes three claims. First, the Fregean reading is precise and literal, not analogical. Second, the fibration structure is already present in any vault with consistent frontmatter — it does not need to be imposed. Third, the emergent structure can be examined to detect divergence, identify implicit schema, and query relational configurations, all without external ontology or validation.</p> <h2 id="1-the-frontmatter-triple">1. The Frontmatter Triple</h2> <h3 id="11-freges-unsaturated-predicate">1.1 Frege&rsquo;s Unsaturated Predicate</h3> <p>Frege&rsquo;s central insight in the <em>Begriffsschrift</em> (1879) is that a predicate is not a label attached to a subject but an unsaturated function that takes an argument and returns a value. &ldquo;Is red&rdquo; is not a property sitting on a shelf; it is an incomplete expression that needs an object to become a complete thought. The expression <span class="katex"><math xmlns="http://www.w3.org/1998/Math/MathML"><semantics><mrow><mi>f</mi><mo stretchy="false">(</mo><mi>x</mi><mo stretchy="false">)</mo></mrow><annotation encoding="application/x-tex">f(x)</annotation></semantics></math></span> is <span class="katex"><math xmlns="http://www.w3.org/1998/Math/MathML"><semantics><mrow><mi>f</mi></mrow><annotation encoding="application/x-tex">f</annotation></semantics></math></span> applied to <span class="katex"><math xmlns="http://www.w3.org/1998/Math/MathML"><semantics><mrow><mi>x</mi></mrow><annotation encoding="application/x-tex">x</annotation></semantics></math></span>, where <span class="katex"><math xmlns="http://www.w3.org/1998/Math/MathML"><semantics><mrow><mi>f</mi></mrow><annotation encoding="application/x-tex">f</annotation></semantics></math></span> is incomplete without <span class="katex"><math xmlns="http://www.w3.org/1998/Math/MathML"><semantics><mrow><mi>x</mi></mrow><annotation encoding="application/x-tex">x</annotation></semantics></math></span> (Heck 2003).</p> <p>This incompleteness is constitutive. A predicate with no argument is not a predicate in a weaker sense — it is not a predicate at all. It is a gap waiting to be filled.</p> <h3 id="12-yaml-frontmatter-as-function-application">1.2 YAML Frontmatter as Function Application</h3> <p>Consider a markdown file with frontmatter:</p> <div class="highlight"><pre tabindex="0" style="color:#f8f8f2;background-color:#272822;-moz-tab-size:4;-o-tab-size:4;tab-size:4;-webkit-text-size-adjust:none;"><code class="language-yaml" data-lang="yaml"><span style="display:flex;"><span>--- </span></span><span style="display:flex;"><span><span style="color:#f92672">authors</span>: <span style="color:#ae81ff">emsenn</span> </span></span><span style="display:flex;"><span><span style="color:#f92672">extends</span>: <span style="color:#ae81ff">legibility</span> </span></span><span style="display:flex;"><span>--- </span></span></code></pre></div><p>The file itself is a resource — a saturated object with a path, a name, content. Each frontmatter key is an unsaturated predicate: <code>authors</code> is incomplete until applied to both a subject (this file) and an object (emsenn). The frontmatter block is a set of function applications:</p> <span class="katex"><math xmlns="http://www.w3.org/1998/Math/MathML" display="block"><semantics><mrow><mtext mathvariant="monospace">authors</mtext><mo stretchy="false">(</mo><mtext>this-file</mtext><mo stretchy="false">)</mo><mo>=</mo><mtext mathvariant="monospace">emsenn</mtext></mrow><annotation encoding="application/x-tex">\texttt{authors}(\text{this-file}) = \texttt{emsenn}</annotation></semantics></math></span><p> </p> <span class="katex"><math xmlns="http://www.w3.org/1998/Math/MathML" display="block"><semantics><mrow><mtext mathvariant="monospace">extends</mtext><mo stretchy="false">(</mo><mtext>this-file</mtext><mo stretchy="false">)</mo><mo>=</mo><mtext mathvariant="monospace">legibility</mtext></mrow><annotation encoding="application/x-tex">\texttt{extends}(\text{this-file}) = \texttt{legibility}</annotation></semantics></math></span><p>This is Frege&rsquo;s structure, implemented. The file is the argument. The key is the function. The value is the output. There is no metaphor here — the YAML parser performs the function application when it reads the file.</p> <h3 id="13-relation-to-rdf">1.3 Relation to RDF</h3> <p>The Resource Description Framework (W3C 2024) structures knowledge as triples: (subject, predicate, object). RDF was designed to implement Berners-Lee&rsquo;s vision of a machine-readable semantic web (Berners-Lee et al. 2001). The frontmatter triple is an RDF triple in which:</p> <ul> <li>The subject is always the file (identified by path or filename)</li> <li>The predicate is always a YAML key</li> <li>The object is always a YAML value</li> </ul> <p>This is a restricted form of RDF — the subject is implicit (always the file) rather than explicitly named. But the restriction is a feature: it means every markdown file is automatically a set of triples without requiring the author to write RDF syntax. The frontmatter format is RDF for people who don&rsquo;t know RDF.</p> <h2 id="2-predicate-fibers">2. Predicate Fibers</h2> <h3 id="21-definition">2.1 Definition</h3> <p>Let <span class="katex"><math xmlns="http://www.w3.org/1998/Math/MathML"><semantics><mrow><mi>V</mi></mrow><annotation encoding="application/x-tex">V</annotation></semantics></math></span> be a vault: a set of markdown files, each with YAML frontmatter. Let <span class="katex"><math xmlns="http://www.w3.org/1998/Math/MathML"><semantics><mrow><mi>K</mi></mrow><annotation encoding="application/x-tex">K</annotation></semantics></math></span> be the set of all keys that appear in any file&rsquo;s frontmatter. For each key <span class="katex"><math xmlns="http://www.w3.org/1998/Math/MathML"><semantics><mrow><mi>k</mi><mo>∈</mo><mi>K</mi></mrow><annotation encoding="application/x-tex">k \in K</annotation></semantics></math></span>, define the <strong>fiber</strong> <span class="katex"><math xmlns="http://www.w3.org/1998/Math/MathML"><semantics><mrow><msub><mi>F</mi><mi>k</mi></msub></mrow><annotation encoding="application/x-tex">F_k</annotation></semantics></math></span> as the set of all values that <span class="katex"><math xmlns="http://www.w3.org/1998/Math/MathML"><semantics><mrow><mi>k</mi></mrow><annotation encoding="application/x-tex">k</annotation></semantics></math></span> takes across all files in <span class="katex"><math xmlns="http://www.w3.org/1998/Math/MathML"><semantics><mrow><mi>V</mi></mrow><annotation mathvariant="normal">∃</mi><mi>f</mi><mo>∈</mo><mi>V</mi><mtext> such that </mtext><mi>k</mi><mo stretchy="false">(</mo><mi>f</mi><mo stretchy="false">)</mo><mo>=</mo><mi>v</mi><mo have <code>authors: emsenn</code> and 3 files have <code>authors: claude</code>, then:</p> <span class="katex"><math xmlns="http://www.w3.org/1998/Math/MathML" \texttt{claude}\}</annotation></semantics></math></span><p>If 15 files use <code>extends:</code> with various values:</p> <span class="katex"><math xmlns="http://www.w3.org/1998/Math/MathML" \texttt{frege-predicate-logic}, \ldots\}</annotation></semantics></math></span><p>Each fiber is a set of tokens. The fibers are indexed by predicate keys. The total structure — all fibers, indexed by all keys — is the vault&rsquo;s <strong>predicate fibration</strong>.</p> <h3 id="22-what-the-fibration-contains">2.2 What the Fibration Contains</h3> <p>The predicate fibration contains exactly the relational structure that the vault&rsquo;s authors have asserted through frontmatter. No more, no less. It does not contain the content of the files, only the claims their frontmatter makes. It does not impose a schema — it reports what schema has emerged from practice.</p> <p>For a vault with rich, consistent frontmatter, the fibration is dense: many keys, many shared values, many files connected through common tokens in the same fibers. For a vault with sparse or inconsistent frontmatter, the fibration is thin: few connections, many singletons.</p> <h3 id="23-relation-to-categorical-logic">2.3 Relation to Categorical Logic</h3> <p>In Lawvere&rsquo;s hyperdoctrine framework (Lawvere 1969), a predicate logic over a base category <span class="katex"><math xmlns="http://www.w3.org/1998/Math/MathML"><semantics><mrow><mi mathvariant="script">C</mi></mrow><annotation encoding="application/x-tex">\mathcal{C}</annotation></semantics></math></span> assigns to each context <span class="katex"><math xmlns="http://www.w3.org/1998/Math/MathML"><semantics><mrow><mi>c</mi><mo>∈</mo><mi mathvariant="script">C</mi></mrow><annotation encoding="application/x-tex">c \in \mathcal{C}</annotation></semantics></math></span> a lattice of propositions <span class="katex"><math xmlns="http://www.w3.org/1998/Math/MathML"><semantics><mrow><mi>P</mi><mo stretchy="false">(</mo><mi>c</mi><mo stretchy="false">)</mo></mrow><annotation encoding="application/x-tex">P(c)</annotation></semantics></math></span>. The assignment <span class="katex"><math xmlns="http://www.w3.org/1998/Math/MathML"><semantics><mrow><mi>P</mi><mo>:</mo><msup><mi mathvariant="script">C</mi><mrow><mi>o</mi><mi>p</mi></mrow></msup><mo>→</mo><mrow><mi mathvariant="bold">H</mi><mi mathvariant="bold">e</mi><mi mathvariant="bold">y</mi><mi mathvariant="bold">t</mi><mi mathvariant="bold">A</mi><mi mathvariant="bold">l</mi><mi mathvariant="bold">g</mi></mrow></mrow><annotation encoding="application/x-tex">P: \mathcal{C}^{op} \to \mathbf{HeytAlg}</annotation></semantics></math></span> is a functor from contexts to Heyting algebras. This is a fibration: each context gets its own fiber of predicates, and these fibers are connected by reindexing maps (substitution).</p> <p>The predicate fibration of a vault is a degenerate case. The base category is the discrete set <span class="katex"><math xmlns="http://www.w3.org/1998/Math/MathML"><semantics><mrow><mi>K</mi></mrow><annotation encoding="application/x-tex">K</annotation></semantics></math></span> of predicate keys (no morphisms between keys). Each fiber <span class="katex"><math xmlns="http://www.w3.org/1998/Math/MathML"><semantics><mrow><msub><mi>F</mi><mi>k</mi></msub></mrow><annotation encoding="application/x-tex">F_k</annotation></semantics></math></span> is a set (not a Heyting algebra — there is no implication structure within a fiber). There are no reindexing maps because the base is discrete.</p> <p>This is the simplest possible fibration. It is worth naming precisely because it already exists in every vault with frontmatter, and because the degenerate structure can be enriched:</p> <ul> <li>If values within a fiber are partially ordered (e.g., <code>status: draft</code> &lt; <code>status: review</code> &lt; <code>status: published</code>), the fiber becomes a poset, and with it a Heyting algebra.</li> <li>If keys are related (e.g., <code>extends</code> and <code>extended-by</code> are dual), morphisms appear in the base, and the fibration acquires reindexing maps.</li> <li>If files are organized into directories that provide context, the base category gains hierarchical structure.</li> </ul> <p>Each enrichment moves the vault closer to a full hyperdoctrine without requiring the vault&rsquo;s author to know what a hyperdoctrine is.</p> <h2 id="3-emergent-schema">3. Emergent Schema</h2> <h3 id="31-schema-first-vs-schema-free">3.1 Schema-First vs. Schema-Free</h3> <p>Traditional knowledge representation is schema-first: define an ontology, then populate it with instances. RDF/OWL works this way — you define classes and properties, then assert triples that conform to the schema. Violations are errors.</p> <p>The predicate fibration inverts this. The vault&rsquo;s authors write frontmatter as they see fit. The fibration reports what has emerged. If 800 files assert <code>authors: emsenn</code> and one file asserts <code>authors: emsen</code>, the fibration does not call this an error. It reports a divergence: one value in the <code>authors</code> fiber that appears once where another appears 800 times.</p> <h3 id="32-divergence-detection">3.2 Divergence Detection</h3> <p>A <strong>divergence</strong> in a fiber is a value that appears rarely relative to the fiber&rsquo;s dominant values. Divergences may indicate:</p> <ul> <li>Typos (<code>emsen</code> for <code>emsenn</code>)</li> <li>Legitimate exceptions (a file with a different author)</li> <li>Evolving conventions (an old key being replaced by a new one)</li> </ul> <p>The fibration cannot distinguish between these. A human or agent must interpret the divergence. But the fibration makes the divergence <em>visible</em> — it surfaces the question &ldquo;is this intentional?&rdquo; without needing a schema to enforce an answer.</p> <h3 id="33-implicit-schema-recovery">3.3 Implicit Schema Recovery</h3> <p>The dominant patterns within fibers constitute the vault&rsquo;s implicit schema. If every file with <code>type: paper</code> also has <code>authors:</code> and <code>status:</code>, then <code>paper</code> implicitly requires <code>authors</code> and <code>status</code> — not because a schema says so, but because practice has determined it. This implicit schema can be extracted, examined, and formalized after the fact, or it can remain implicit and serve as a soft constraint.</p> <h2 id="4-what-this-is-not">4. What This Is Not</h2> <p>This paper does not claim that YAML frontmatter is a replacement for RDF, OWL, or formal ontology engineering. Those systems provide inference, constraint checking, and reasoning that the predicate fibration does not. The claim is narrower: the Fregean structure is already there in frontmatter, the fibration is already there in any vault, and examining the emergent structure is useful even without external tooling.</p> <p>This paper also does not claim novelty for the Fregean reading of subject-predicate-object triples — that connection is well-established in the RDF literature. The contribution is recognizing that YAML frontmatter in markdown vaults is a concrete, widespread, practical instantiation of this structure, and that the fibration it generates can be examined as a mathematical object.</p> <h2 id="5-practical-consequences">5. Practical Consequences</h2> <h3 id="51-for-knowledge-workers">5.1 For Knowledge Workers</h3> <p>Any vault with consistent frontmatter already has a queryable knowledge graph. The graph does not need to be extracted into a database — it can be read directly from the YAML. The predicate fibration tells you: what predicates exist, what values each predicate takes, which files share values, and where conventions diverge.</p> <h3 id="52-for-ai-agents">5.2 For AI Agents</h3> <p>AI agents that read vaults (via CLAUDE.md, AGENTS.md, or Model Context Protocol) already consume frontmatter. The fibration gives them a way to understand the vault&rsquo;s structure without reading every file: examine the fibers, identify the dominant patterns, and use those patterns to navigate the vault&rsquo;s content.</p> <h3 id="53-for-vault-design">5.3 For Vault Design</h3> <p>If you design frontmatter with the fibration in mind — using short, reusable tokens as values, using predicate keys that carry relational weight (defines, extends, requires, acts-on, produces, contrasts-with) rather than flat labels (tags) — you build a denser, more traversable graph. Each value becomes a node that connects every file that uses it. Each predicate becomes a typed edge. The vault becomes a knowledge base, not by importing an external ontology, but by writing careful frontmatter.</p> <h2 id="6-conclusion">6. Conclusion</h2> <p>YAML frontmatter is Fregean predicate logic implemented in plain text. Each key is an unsaturated function. Each file is an argument. Each value is the output. Across a vault, shared keys generate fibers of values that constitute a predicate fibration — a degenerate hyperdoctrine whose structure emerges from practice rather than being imposed by schema. This structure is already present in every vault with frontmatter. Examining it yields knowledge about the vault&rsquo;s implicit conventions, its divergences, and its relational topology, without requiring any formalism the author did not already use.</p> <h2 id="references">References</h2> <ul> <li>Berners-Lee, T., Hendler, J., &amp; Lassila, O.
