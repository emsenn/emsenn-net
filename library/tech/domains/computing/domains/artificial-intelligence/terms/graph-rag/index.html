GraphRAG

2026-03-29 A retrieval-augmented generation pattern that retrieves from a knowledge graph rather than a flat vector store — using entity relationships to answer multi-hop queries that standard RAG cannot.

<p>GraphRAG is a <a href="./retrieval-augmented-generation.md" class="link-internal">retrieval-augmented generation</a> pattern that retrieves from a knowledge graph rather than a flat vector store. Standard RAG embeds document chunks as vectors and retrieves by semantic similarity. GraphRAG extracts entities and relationships from documents, builds a graph connecting them, and traverses that graph to answer queries that require reasoning across multiple documents or multiple hops of relationship.</p> <p>Microsoft Research introduced the term and architecture in &ldquo;From Local to Global: A Graph RAG Approach to Query-Focused Summarization&rdquo; (Edge et al., 2024). The core insight: standard RAG retrieves passages similar to the query, but many questions require synthesizing information scattered across passages that are not individually similar to the query. &ldquo;What are the main themes in this dataset?&rdquo; requires global understanding. &ldquo;How does X relate to Y through Z?&rdquo; requires following a chain of relationships. Standard RAG retrieves individual chunks; GraphRAG retrieves subgraphs.</p> <h3 id="how-it-works">How it works</h3> <ol> <li><strong>Entity and relationship extraction.</strong> An LLM processes source documents and extracts entities (people, concepts, organizations, terms) and relationships between them (uses, extends, contradicts, cites).</li> <li><strong>Graph construction.</strong> Extracted entities become nodes; relationships become edges. Community detection algorithms (Leiden, Louvain) identify clusters of related entities.</li> <li><strong>Indexing.</strong> Each community gets a summary. Entities and relationships get <a href="./embedding.md" class="link-internal">embeddings</a>. The graph and its summaries are stored in a graph database.</li> <li><strong>Retrieval.</strong> A query is matched against entity embeddings, community summaries, or both. The system traverses the graph from matched entities to find relevant subgraphs — chains of relationships that connect query-relevant concepts.</li> <li><strong>Generation.</strong> Retrieved subgraphs and community summaries are injected into the model&rsquo;s <a href="./context-window.md" class="link-internal">context window</a>. The model generates its response grounded in the graph structure.</li> </ol> <h3 id="performance-characteristics">Performance characteristics</h3> <p>A systematic evaluation (arXiv 2502.11371, February 2025) found:</p> <ul> <li>GraphRAG delivers <strong>1.5x better accuracy overall</strong> and <strong>2x better accuracy on multi-hop queries</strong> compared to standard RAG</li> <li>For schema-heavy enterprise queries, GraphRAG outperforms vector RAG by <strong>3.4x</strong> (Diffbot/FalkorDB KG-LM Benchmark)</li> <li>GraphRAG achieves <strong>90% hallucination reduction</strong> for schema-heavy queries (FalkorDB 2025)</li> <li>GraphRAG shows <strong>13.4% lower accuracy</strong> on simple single-hop queries compared to standard RAG — it adds overhead without benefit when the query matches a single passage</li> <li>Vector RAG accuracy degrades to <strong>0%</strong> as entity count per query exceeds 5; GraphRAG sustains stable performance with 10+ entities</li> <li>GraphRAG incurs <strong>2.4x higher latency</strong> than standard RAG</li> </ul> <h3 id="lightrag">LightRAG</h3> <p>LightRAG (October 2024, peer-reviewed at EMNLP 2025 Findings) achieves comparable accuracy to GraphRAG with dramatically lower cost: <strong>6,000x token reduction</strong> (sub-100 tokens vs. 610,000), ~30% latency reduction, and 65-80% cost savings for high-volume processing. It uses dual-level graph retrieval — local entity-level and global community-level — without the expensive community summarization step.</p> <h3 id="when-to-use-graphrag">When to use GraphRAG</h3> <p>Use GraphRAG when queries require reasoning across relationships between 5+ entities, multi-hop traversal, or global summarization. Use standard <a href="./retrieval-augmented-generation.md" class="link-internal">RAG</a> for simple factual lookups and single-passage retrieval. Use LightRAG as a middle ground when GraphRAG&rsquo;s cost is prohibitive. Hybrid approaches — standard RAG for simple queries, GraphRAG for relational queries — consistently outperform either alone.</p> <p>For knowledge repositories with structured metadata (semantic frontmatter with <code>uses</code>, <code>related</code>, <code>extends</code> fields), the metadata already encodes a graph. Building a graph index from existing frontmatter is cheaper than extracting entities from prose because the entities and relationships are already declared.</p> <h2 id="related-terms">Related terms</h2> <ul> <li><a href="./retrieval-augmented-generation.md" class="link-internal">Retrieval-augmented generation</a> — the pattern GraphRAG extends</li> <li><a href="./embedding.md" class="link-internal">Embedding</a> — the representations used for entity and query matching</li> <li><a href="./hallucination.md" class="link-internal">Hallucination</a> — the failure mode GraphRAG reduces on relational queries</li> <li><a href="../domains/agents/terms/agent-memory.md" class="link-internal">Agent memory</a> — graph-structured memory as a persistence strategy</li> </ul>
