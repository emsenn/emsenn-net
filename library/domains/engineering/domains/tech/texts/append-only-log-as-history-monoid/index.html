Append-Only Logs Are History Monoids

2026-04-04 Append-only event logs are trace monoids (history monoids) — formal algebraic objects where independent events commute and dependent events do not. This structure is not a metaphor; it is the mathematical identity of event-sourced systems.

An append-only event log — the kind used in event sourcing, write-ahead logs, and ledger systems — is not merely a sequence of records. It is a trace monoid: a free partially commutative monoid where independent events may be reordered without changing the system’s state, but dependent events must remain in sequence.

This is not an analogy. The algebraic structure of the log is the trace monoid. Recognizing this turns operational questions (Can I parallelize these events? Does replay order matter? What is the canonical form of this history?) into algebraic computations with known solutions.

¶The Free Monoid of Events

Any append-only log over an alphabet Σ of event types forms a free monoid Σ*: the set of all finite sequences of events, with concatenation as the operation and the empty sequence as identity. This is the structure you get before imposing any equivalences — every distinct sequence is a distinct history.

But not all reorderings change the outcome. If event a modifies account X and event b modifies account Y, the final state is the same whether a precedes b or b precedes a. The events are independent.

¶The Independence Relation

Define an independence relation I ⊆ Σ × Σ where (a, b) ∈ I means events a and b commute: applying them in either order yields the same result. The complement D = (Σ × Σ) \ I is the dependence relation: pairs of events whose order matters.

For event logs, the natural independence relation is:

  • Events on different subjects are independent (they touch disjoint state).
  • Events on the same subject are dependent (they modify shared state, and order determines the outcome).

This is exactly the structure identified by Mazurkiewicz (1977) and developed by Diekert and Rozenberg (1995) as the theory of trace monoids.

¶The Trace Monoid

The trace monoid M(Σ, I) is the quotient of the free monoid Σ* by the congruence generated by I:

ab ~ ba  whenever (a, b) ∈ I

Two sequences are equivalent (represent the same trace) if one can be obtained from the other by swapping adjacent independent events. Each equivalence class — each trace — represents a partial order of events: dependent events are ordered, independent events are unordered.

The trace monoid has concrete computational properties:

  1. Normal form: Every trace has a canonical representative — the lexicographically first sequence in its equivalence class. This gives a canonical serialization of any history.

  2. Parallelism: The Foata normal form (Cartier & Foata, 1969) factors each trace into a sequence of steps — maximal sets of mutually independent events. Each step can execute in parallel. The number of steps is the height of the trace, which is the minimum sequential execution time.

  3. Prefix order: Traces are partially ordered by prefix. The prefix order is the causal history: trace s is a prefix of trace t if every event in s causally precedes the additional events in t.

¶Event Sourcing and Trace Equivalence

In event-sourced systems (as described by Fowler, 2005, and codified in systems like EventStore and Kafka), the log is the source of truth. The current state is derived by replaying the log from the beginning.

The trace monoid structure tells you exactly when replay order matters:

  • Replay is deterministic up to trace equivalence: Any two serializations of the same trace produce the same final state. You can reorder independent events freely during replay.
  • Compaction preserves traces: Removing redundant events (e.g., superseded updates to the same subject) is safe if and only if the removed events are in the same trace equivalence class as shorter sequences.
  • Concurrent appends commute: Two processes appending independent events to the log produce the same trace regardless of which append lands first. This is why event logs tolerate concurrent writers on disjoint subjects without coordination.

¶The Dependence Graph

Each trace corresponds to a dependence graph: a directed acyclic graph where nodes are events and edges connect dependent events in causal order. The dependence graph is the Hasse diagram of the partial order defined by the trace.

Lamport’s (1978) happened-before relation on distributed events is exactly the dependence order of the trace monoid when Σ is the set of message sends and receives and I relates events at different processes that do not exchange messages. The connection between Mazurkiewicz traces and Lamport causality was made explicit by Pratt (1986) and Winskel (1986).

¶Why This Matters

Recognizing event logs as trace monoids turns design questions into algebraic ones:

  • “Can these events be parallelized?” → Are they independent in I? Compute the Foata normal form.
  • “Is this replay correct?” → Is the replay sequence in the same trace equivalence class as the original log?
  • “What is the minimum latency for processing this history?” → What is the height of the trace (number of Foata steps)?
  • “Can I merge these two logs?” → Are they prefixes of a common trace? Compute the join in the prefix order.

These are solved problems in trace monoid theory, with efficient algorithms (Diekert & Rozenberg, 1995).
