Tool Call

2026-03-30 The atomic unit of agent-environment interaction — a structured invocation of a named function with parameters, executed by the runtime and returning a result to the model.

A tool call is the atomic unit of interaction between an AI agent and its environment. The model generates a structured request — a tool name and a set of parameters conforming to a schema — and the host runtime executes the request against the environment and returns a result. The model then reasons about the result and may issue further tool calls. This call-observe-reason-call loop is the basic rhythm of agentic behavior.

A tool call is distinct from tool use as a capability: tool use is the model’s ability to generate tool calls; a tool call is a single instance of that ability being exercised. The relationship is analogous to “language” (capability) versus “utterance” (instance).

The structure of a tool call is fixed across providers and protocols:

  1. Name — identifies which tool to invoke (e.g., Read, Bash, Edit)
  2. Parameters — a JSON object conforming to the tool’s input schema
  3. Execution — the runtime (not the model) performs the action
  4. Result — the output is returned to the model as a message

The model never executes a tool call directly. The runtime mediates: it validates the parameters against the schema, checks permissions, executes the call, and returns the result. This separation between intent (the model’s generated request) and execution (the runtime’s action) is what makes permission models and hooks possible — they intercept the call at the runtime layer.

In Claude Code, tool calls are the events that hooks attach to: PreToolUse fires before execution, PostToolUse fires after. In the Model Context Protocol, tool calls are JSON-RPC requests from the MCP client to the MCP server. In both cases, the structure is the same: name, parameters, execute, result.
