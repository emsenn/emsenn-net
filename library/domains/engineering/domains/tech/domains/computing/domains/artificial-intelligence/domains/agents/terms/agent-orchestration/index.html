2026-03-29 The coordination layer that determines which steps an agentic system executes in what order, under what conditions, and with what state.

Agent orchestration is the coordination of one or more agentic systems through a defined execution topology. The topology determines which steps execute in what order, under what conditions, and with what state. Orchestration sits between the raw tool use primitive and the goal the system is pursuing.

The simplest orchestration is a linear chain: call tool A, feed the result to tool B, return. The most complex involve graph-structured workflows with conditional branching, parallel execution, checkpointing, rollback, human-in-the-loop pauses, and dynamic subgraph composition.

Four design decisions define any orchestration:

State management. How is the running state represented, persisted, and recovered after failure? The answer ranges from “conversation history in memory” (volatile, lost on crash) to “typed state object checkpointed to a database at every step” (durable, resumable, debuggable). Durable state is what separates an orchestration from a script — the system can crash and pick up where it left off.

Control flow. Is the execution path predefined or model-directed? Predefined paths are workflows. Model-directed paths are agents. Between them is a spectrum: prompt chaining (fixed sequence), routing (model classifies, fixed handlers), parallelization (independent subtasks), orchestrator-workers (model delegates), and evaluator-optimizer (model judges its own output). Each step gives the model more control over what happens next.

Communication. In multi-agent configurations, how do agents discover each other and exchange messages? MCP standardizes agent-to-tool connections. A2A standardizes agent-to-agent communication. Within a single runtime, frameworks use function calls, shared state, or message queues.

Observability. Can the orchestration be inspected, replayed, and debugged? This is not optional — without it, failures in multi-step workflows are opaque. Checkpoint-based systems enable time-travel debugging (restore any prior state). Event-sourced architectures enable deterministic replay.

Conflict resolution. When agents produce contradictory outputs or propose actions with safety implications, what reconciles them? Patterns include proposal-based execution (an agent proposes an action with reasoning, a validator checks preconditions before any side effect), multi-agent consensus (a critical decision requires independent agreement from two or more agents before triggering), and threshold-based escalation to human review when confidence falls below a bound. This decision becomes load-bearing in safety-critical deployments — industrial control, medical workflows — where a wrong action is more expensive than no action.

The orchestration/choreography distinction. Orchestration is central: one process directs others. Choreography is decentralized: participants follow a shared protocol independently. The W3C Web Services Choreography Description Language (2005) formalized this distinction; Milner’s π-calculus (1999) gives the process-algebraic foundation. Most practical systems mix both — a Claude Code session orchestrates subagents (centralized dispatch) while those subagents follow shared conventions encoded in skills and rules (decentralized protocol adherence).

No existing framework provides formal compositional guarantees — region-restricted execution, provable non-interference between concurrent agents, or skill admissibility conditions. These remain open problems.
