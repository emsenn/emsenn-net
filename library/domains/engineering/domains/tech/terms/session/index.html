2026-04-01 A bounded interval of interaction between a user and a system, with a beginning, an end, and state that persists within but not automatically across the boundary.

A session is a bounded interval of interaction between a user and a system. It has a beginning (the user initiates contact), an end (the user disconnects or the system’s resources expire), and state that persists within the session but does not automatically survive its boundary. When a session ends, its state is lost unless something explicitly saves it.

The concept comes from timesharing systems. In Multics (Fernando José Corbató, 1965), multiple users shared a single computer by taking turns. Each user’s session began when they logged in and ended when they logged out. The operating system maintained each user’s state — open files, running processes, environment variables — for the duration of the session and cleaned it up afterward. The session was the unit of resource allocation: the system gave each session a share of processor time, memory, and storage.

The web inherited the concept but lost the statefulness. HTTP (Tim Berners-Lee, 1991) is stateless — each request is independent, carrying no memory of previous requests. Lou Montulli invented the HTTP cookie in 1994 at Netscape specifically to restore session state to a stateless protocol. A session cookie links successive HTTP requests into a single session, allowing the server to maintain state (shopping cart contents, login status, form progress) across what would otherwise be disconnected interactions.

In agentic systems, a session is a single conversation between a user and an agent. The agent’s context — everything it knows, everything it has read, every decision it has made — exists within the session. When the session ends, that context is gone. This is why agentic systems need explicit mechanisms for cross-session persistence: memory files that survive the session boundary, plans that can be resumed, session logs that record what happened. Without these mechanisms, every session starts from zero.

The session boundary is the fundamental constraint on agentic work. Within a session, an agent can hold a complex mental model, track multiple threads, and maintain coherence across hundreds of actions. Across sessions, it cannot — unless the system provides persistence mechanisms that externalize session state into durable artifacts. The quality of cross-session work depends entirely on the quality of these mechanisms: how much of the session’s state can be captured, how faithfully it can be restored, and how quickly the agent can reconstruct a working context from saved artifacts.
