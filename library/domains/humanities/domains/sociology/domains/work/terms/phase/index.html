Phase

2026-04-04 A distinct stage within a lifecycle — a bounded period during which an entity has a particular character and defined entry and exit conditions.

A phase is a distinct stage within a lifecycle — a bounded period during which an entity has a particular character and defined entry and exit conditions. A phase is one period in the temporal arc of an entity’s existence.

A phase is a state understood temporally. A state is a snapshot at a single moment — the configuration of all conditions defining a thing right now. A phase is a state that persists over time. The “in review” phase takes hours or days. The “draft” phase has a different character than the “published” phase. The phase is not just the configuration but the duration of that configuration. Three conditions define a phase: entry conditions (what must be true to enter), the character of the phase itself (what the entity is like while in this phase), and exit conditions (what must be true to leave).

Phases compose into lifecycles. A task lifecycle has phases: open → active → completed (or failed). A plan lifecycle has phases: draft → active → completed → archived. A project lifecycle has phases: initiation → planning → execution → monitoring and control → closing. The ordering of phases may be strictly linear (waterfall: requirements → design → implementation → testing, each follows the previous without deviation) or flexible (agile: phases can overlap, loop, or branch based on conditions). But even in flexible orderings, the phases constrain which transitions are valid. A task cannot move backward from completed to active without a deliberate reopening event. A plan cannot enter the archived phase until it is completed. These constraints encode the logical structure of what it means to be that kind of entity.

Phase transitions are events — discrete moments that change the entity’s phase. A task moves from “open” to “active” when someone begins work — that transition is an event. A plan moves from “draft” to “active” when it is authorized — that event marks the transition. Each event moves an entity from one phase to another. The event is instantaneous; the phase is the interval between events.

This pattern appears across domains because any entity existing in time must pass through phases. A plant has developmental phases: seed dormancy → germination → vegetative growth → flowering → fruiting → senescence. A fungal mycelium has ecological phases: dispersal → colonization → growth → fruiting → decline. A research inquiry has phases: question formation → literature review → design → data collection → analysis → reporting. Phases are not unique to project management or work systems — they are fundamental to any ordered temporal process.

The phase concept is distinct from related ideas in the work taxonomy. A milestone marks the boundary between phases — it is a recognizable state achieved at the transition point. A process is work with a fully determined sequence of phases — the input determines the exact phases that will follow. A procedure is work with branching phases — which phase comes next depends on a decision made during execution. An inquiry is work with emergent phases — the phases themselves are discovered during execution. But all four types have phases; what differs is whether the phases are predetermined, branching, or emergent.

Phases structure how work duration is understood and allocated. A capacity plan must account for which phases are resource-intensive and which are lighter. A schedule assigns phases to calendar time. A workflow chains entities’ phases together so that one entity’s transition (a phase exit in one task) triggers another entity to enter a new phase. The phase is the unit of time that makes all temporal coordination possible.
