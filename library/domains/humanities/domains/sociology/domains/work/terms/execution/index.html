Execution

2026-04-07 The sequence of acts that advances a work entity from assigned toward completed. Where plans meet reality — where assumptions are tested, errors occur, artifacts are produced, and the system's resilience is measured.

Execution is the stage of the operations loop where work actually happens. A task has been assigned to an agent. Execution is the sequence of acts that advance the task’s state from assigned toward completed.

¶What execution produces

Execution produces three things:

  1. State changes — the task moves through states (assigned → in-progress → completed, or → blocked, or → failed). Each transition is an act that changes the work entity’s state.

  2. Artifacts — the outputs of the work. A research task produces a library text. An enrichment task produces modified frontmatter. A build task produces compiled code. The artifact is the ground term — the fully specified output of the fully specified work unit.

  3. Events — the timestamped record of what happened. Every act during execution should produce an event. The event sequence IS the audit trail. It is not a separate logging system — it is the natural consequence of taking acts seriously as entities with temporal coordinates.

¶The TOTE unit

Miller, Galanter, and Pribram (Plans and the Structure of Behavior, 1960) describe the basic unit of execution as a TOTE: Test-Operate-Test-Exit. The agent tests the current state against the goal. If the state doesn’t match, the agent operates (performs an act). Then tests again. When the state matches the goal, the agent exits. This is the same structure as the settlement cycle: compose (⊙) the current state with the goal, check (⇒) whether they agree. If ⊥ or ⊤, operate. If t, done.

¶Execution and work types

The work type determines what execution looks like:

  • Process execution is deterministic — same input, same output, no branching. Run the script. Apply the template.
  • Procedure execution branches on conditions — evaluate the state, take the appropriate path. Follow the checklist, adapting to what you find.
  • Derivation execution produces a value through inference — the operation is known but the result is not predetermined. Compute the metric. Write the finding.
  • Inquiry execution discovers its own structure — neither the operation nor the result is predetermined. Research the question. The path emerges during the walk.

¶Execution and errors

Reason (Human Error, 1990) distinguishes slips (correct intention, wrong action), lapses (correct intention, forgotten action), and mistakes (wrong intention). Each type of error produces a different kind of event:

  • A slip produces a state change that doesn’t match the intended change — detected by tracking
  • A lapse produces no state change where one was expected — detected by tracking as absence
  • A mistake produces the intended state change but the intention was wrong — detected by review

The saga pattern (Garcia-Molina and Salem, 1987) handles execution failures in multi-step workflows: each step has a compensating action that undoes it. If step 3 fails, compensate steps 2 and 1. This is how the system recovers from errors without requiring the entire workflow to be atomic.

¶The algebraic reading

In the algebraic structure of work, execution is where open terms become ground terms. An assigned task is an open term — operate(agent, ?, ?) — with the operation bound (the agent knows what to do) but some arguments still free (the specific content, the specific context). Execution binds the remaining variables through the acts performed. The completed task is a ground term: operate(agent, specific-input, specific-output).

The settlement path position during execution: the task moves from whatever work type it started as toward process. An inquiry-type task during execution may settle some of its meaning (discovering what it’s about), moving toward derivation. A derivation-type task during execution may settle its execution path, moving toward procedure. Execution IS settlement happening in real time.
