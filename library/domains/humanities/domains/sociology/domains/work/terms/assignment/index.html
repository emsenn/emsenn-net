Assignment

2026-04-03 The act of creating a relation between a work unit and an agent — routing specific work to a specific doer with defined authority and capacity constraints.

An assignment is the act of creating a relation between a work unit and an agent. When a task is assigned, it gains an assignee — an agent who is now responsible for advancing it. The assignment is the central act of the dispatch stage.

An assignment is not just “telling someone to do something.” It is a structured act that checks three conditions before it can succeed:

Capability — the agent can do this type of work. A code review task requires an agent with code review skills. A term-writing task requires an agent with the writing-terms skill. Capability is a quality of the agent, matched against a requirement of the task.

Authority — the agent is permitted to do this work within this scope. In Hohfeldian terms: the agent has the privilege to perform the work, and the assigner has the power to create the assignment. An agent cannot accept work outside its authority scope. A dispatcher cannot assign work beyond its delegation power.

Capacity — the agent has bandwidth. An agent with 15 tasks in progress should not receive a 16th. Little’s Law governs this: more work in progress means longer cycle time per task. Capacity checking prevents overload.

An assignment creates an event — a timestamped record of who assigned what to whom, under what authority, at what time. The event is part of the audit trail. It cannot be silently undone; reassignment creates a new event.

In multi-agent systems (Gerkey and Matarić 2004), assignments can be instantaneous (decided once) or time-extended (adjusted as conditions change). A settlement-carrier system typically uses time-extended assignment: tasks can be reassigned if agents fail, become unavailable, or if the work changes.

See Dispatch and Delegation Survey for the full theoretical grounding.
