Monads and Comonads

¶Entry conditions

Use monads and comonads when you can:

  • Start from a category (or a poset viewed as a thin category).
  • Identify an endofunctor T : C -> C that “wraps” or “annotates” objects.
  • Ask whether the wrapping is coherent: can it be applied once, twice, or unwrapped, in ways that compose consistently?

¶Definitions

A monad on a category C is a triple (T, eta, mu) where:

  • T : C -> C is an endofunctor,
  • eta : id -> T is a natural transformation called the unit (wrapping),
  • mu : T . T -> T is a natural transformation called the multiplication (flattening),

satisfying:

  • Associativity: mu . T(mu) = mu . mu_T
  • Unit laws: mu . T(eta) = id = mu . eta_T

A comonad on C is the dual: a triple (G, epsilon, delta) where:

  • G : C -> C is an endofunctor,
  • epsilon : G -> id is a natural transformation called the counit (extraction),
  • delta : G -> G . G is a natural transformation called the comultiplication (duplication),

satisfying:

  • Coassociativity: G(delta) . delta = delta_G . delta
  • Counit laws: G(epsilon) . delta = id = epsilon_G . delta

On a poset (H, <=) viewed as a thin category, a comonad simplifies to:

  • G : H -> H monotone,
  • epsilon: G(a) <= a for all a (extraction is deflating),
  • delta: G(a) <= G(G(a)) for all a (duplication is inflating within G),

with coassociativity and counit laws holding as inequalities.

¶Vocabulary (plain language)

  • Monad: wraps objects and flattens double-wrapping. Think of it as a notion of computation or context that can be composed.
  • Comonad: extracts values from context and duplicates context. The dual of a monad — where a monad adds structure, a comonad reads it.
  • Counit (epsilon): extracts the underlying value from a comonadic context. “What is the thing, ignoring its annotation?”
  • Comultiplication (delta): duplicates the annotation. “The traced version of a trace is itself traced.” This enables iterable annotation.

¶Intuition

A monad wraps things and lets you flatten nested wrapping. The canonical example: lists. Wrapping a value gives a singleton list; flattening a list of lists concatenates them.

A comonad extracts things and lets you duplicate context. The canonical example on a poset: a closure operator’s dual. If G(a) is “the part of a that is directly observable,” then epsilon(a) says “what you observe is less than what is there,” and delta(a) says “observing the observation is still observation.”

In the semiotic universe, the trace comonad G records interpretive history. The counit extracts the current semantic value. The comultiplication records that the tracing itself has been traced — the self-referential structure that semiotic interpretation requires.

¶Heyting-comonadic coherence

When G is a comonad on a Heyting algebra H, additional structure becomes possible. G is Heyting-comonadic if it preserves all Heyting operations:

  1. G(a meet b) = G(a) meet G(b)
  2. G(a join b) = G(a) join G(b)
  3. G(a => b) = G(a) => G(b)
  4. G(top) = top, G(bot) = bot

This means tracing does not distort logical structure. The traced version of a conjunction is the conjunction of the traces; the traced version of an implication is the implication of the traces.

This coherence axiom (Axiom 1.2 of the semiotic universe specification) is what makes the trace comonad more than an arbitrary annotation — it ensures that temporal unfolding respects the logic of meaning.

¶Worked example

Let H = {bot, a, b, top} be the diamond lattice where a and b are incomparable, with the Heyting implication defined by the adjunction.

Define G : H -> H as:

  • G(bot) = bot
  • G(a) = bot
  • G(b) = bot
  • G(top) = top

Then:

  • Counit: G(x) <= x for all x. Check: bot <= bot, bot <= a, bot <= b, top <= top. Holds.
  • Comultiplication: G(x) <= G(G(x)) for all x. G(G(bot)) = G(bot) = bot, so bot <= bot. G(G(a)) = G(bot) = bot, so bot <= bot. G(G(top)) = G(top) = top, so top <= top. Holds.
  • Heyting preservation: G(a meet b) = G(bot) = bot = bot meet bot = G(a) meet G(b). G(a join b) = G(top) = top = bot join top… wait, bot join bot = bot, but G(a join b) = G(top) = top. So top != bot. Fails.

This G is a comonad but NOT Heyting-comonadic. The preservation of joins requires G to distribute, which this G does not. The constraint is real: not every comonad on a Heyting algebra preserves the Heyting structure.

¶Common mistakes

  • Confusing monads with comonads. The arrows reverse: unit vs. counit, multiplication vs. comultiplication. Monads wrap; comonads extract.
  • Assuming every comonad on a Heyting algebra preserves Heyting operations. The worked example shows this fails in general. Heyting-comonadic coherence is an additional axiom.
  • Forgetting the counit laws. Coassociativity alone is not enough; the counit laws ensure extraction is well-behaved.

¶Minimal data

  • A category C (or a poset viewed as a thin category).
  • An endofunctor G : C -> C.
  • Natural transformations epsilon : G -> id and delta : G -> G . G.
  • The coassociativity and counit diagrams commute.
  • For Heyting-comonadic coherence: G preserves meets, joins, implication, top, and bottom.
