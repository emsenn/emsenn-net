2026-03-20 A Task is a triple (G, S, P) — a goal G specifying the desired outcome, a set S of subtasks or atomic operations required to achieve G, and a plan P ordering the elements of S under the conditions in which each must execute — constituting a recognizable, purposive unit of activity that decomposes until atomic operations are reached. The defining structure: a task is organized around a goal that defines what counts as its completion; the task's structure is the hierarchical decomposition of the goal into sub-goals, each with its own sub-tasks, terminating in atomic operations. A task is not a procedure (a procedure has an a priori specification; a task's plan is conditional on context) and not a plan (a plan is a set of partially ordered operations; a task is the goal-structured unit of activity that the plan serves).

Task

Formal definition

A Task is a triple (G,S,P)(G, S, P):

(G:Goal,  S:Subtasks∪Operations,  P:Plan(S,G))(G : \mathrm{Goal},\; S : \mathrm{Subtasks} \cup \mathrm{Operations},\; P : \mathrm{Plan}(S, G))

where:

  • GG is the goal — the desired outcome or end-state the task aims to achieve; GG is what makes the task purposive: the task succeeds iff GG is achieved; the goal is not merely a postcondition but a purposive state — a state the agent is attempting to bring about; the goal gives the task its unit-character: a task is bounded by its goal’s achievement
  • SS is the subtask set or operation set — the constituent elements required to achieve GG; at a non-atomic level, S={T1,…,Tn}S = \{T_1, \ldots, T_n\} is a set of sub-tasks, each with their own (Gi,Si,Pi)(G_i, S_i, P_i) structure; at the atomic level, SS consists of primitive operations — elementary actions that cannot be further decomposed; the hierarchy terminates when every leaf is an operation
  • PP is the plan — the partial ordering and conditional structure specifying how elements of SS must be sequenced to achieve GG; PP specifies ordering constraints, conditions under which each subtask applies, and the stopping condition (when GG is deemed achieved); PP may be underspecified — leaving open the choice between multiple valid orderings

Three invariants. (G,S,P)(G, S, P) is a task iff:

  1. Goal-directed: GG is defined before SS and PP — the task’s purpose determines what counts as a subtask and what counts as a valid plan. An operation not connected to GG is not part of the task. This distinguishes a task from a mere sequence: a sequence has no goal; a task has a goal that defines when the sequence is complete and which operations are relevant.

  2. Hierarchical decomposability: each non-atomic subtask Ti∈ST_i \in S is itself a task (Gi,Si,Pi)(G_i, S_i, P_i) where achieving GiG_i makes progress toward GG. The hierarchy terminates: there are no infinite chains of decomposition. The atomic operations at the leaves are the irreducible executions from which higher-level behavior is built.

  3. Plan-goal coherence: executing the operations in SS in the order specified by PP produces a state satisfying GG. A task whose plan does not achieve its goal is a broken task. Coherence is not a syntactic property of PP; it is the semantic requirement that PP’s execution actually achieves GG given the task’s context.

Annett and Duncan: hierarchical task analysis

John Annett and Keith Duncan (Task Analysis and Training Design, 1967, Occupational Psychology) introduced hierarchical task analysis (HTA) as a method for analyzing task structure for training and system design:

An HTA decomposition of a task has:

  • A goal for the whole task
  • A plan specifying the operations and the order/conditions under which they occur
  • An iterative subgoal decomposition: each non-atomic operation is re-analyzed as a sub-goal with its own operations and plan

The stopping criterion — the p×cp \times c rule: stop decomposing when the product of pp (probability that inadequate performance causes failure) and cc (cost of inadequate performance) falls below an acceptable threshold. High p×cp \times c operations require further analysis; low p×cp \times c can be treated as atomic.

Plans in HTA specify conditions under which subtasks execute:

  • Fixed sequence: “Do 1, then 2, then 3”
  • Contingent: “If condition XX, do 1; else do 2”
  • Concurrent: “Do 1 and 2 simultaneously, then do 3 when both complete”
  • Cyclic: “Do 1 repeatedly until condition YY”

HTA is foundational in human factors and cognitive systems engineering for understanding how complex skilled behaviors are organized and how they can fail.

Erol, Hendler, and Nau: hierarchical task networks

Kutluhan Erol, James Hendler, and Dana Nau (HTN Planning: Complexity and Expressivity, 1994; UMCP: A Sound and Complete Procedure for Hierarchical Task-Network Planning, 1994) formalized hierarchical task network (HTN) planning — making precise the distinction between primitive and compound tasks:

A primitive task tpt_p is a task directly executable by an operator (a STRIPS-style action schema): it has no further decomposition. An operator op\mathrm{op} is applicable to tpt_p iff pre(op)\mathrm{pre}(\mathrm{op}) holds in the current state; applying op\mathrm{op} produces effects eff(op)\mathrm{eff}(\mathrm{op}).

A compound task tct_c is a task that must be reduced to primitive tasks via methods. A method m=(tc,prem,Tm)m = (t_c, \mathrm{pre}_m, T_m) for compound task tct_c consists of:

  • prem\mathrm{pre}_m: the precondition under which mm is applicable
  • Tm=({t1,…,tk},≺m)T_m = (\{t_1, \ldots, t_k\}, \prec_m): a task network — a partially ordered set of subtasks that accomplishes tct_c

HTN planning is the process of reducing all compound tasks to primitive tasks via applicable methods:

  1. Select an unresolved compound task tct_c in the current task network
  2. Select an applicable method mm for tct_c (one whose precondition holds)
  3. Replace tct_c with the subtask network TmT_m, inheriting ordering constraints
  4. Repeat until all tasks are primitive

The resulting totally ordered sequence of primitive tasks is the executable plan.

Expressivity: Erol, Hendler, and Nau prove that HTN planning is strictly more expressive than classical (STRIPS-style) planning: any classical planning problem can be encoded as an HTN problem, but not vice versa. HTN planning can directly represent task hierarchies, procedural knowledge, and decomposition constraints that classical planning cannot express without encoding. The complexity: HTN planning with general methods is undecidable; with ground methods (fully instantiated, no variables), it is decidable and EXPTIME-complete.

The primitive/compound distinction is the formal realization of HTA’s leaf/internal-node decomposition: primitive tasks are the leaves of the task hierarchy (corresponding to HTA’s atomic operations at the stopping criterion); compound tasks are the internal nodes (corresponding to HTA’s subgoals requiring further analysis).

Card, Moran, and Newell: GOMS

Stuart Card, Thomas Moran, and Allen Newell (The Psychology of Human-Computer Interaction, 1983) developed GOMS (Goals, Operators, Methods, Selection rules) as a computational cognitive model of routine skilled behavior:

  • Goals: desired states; goals have a hierarchical structure — top-level goals decompose into sub-goals, down to method-executable atomic goals
  • Operators: elementary actions, each with a fixed execution time; the Keystroke Level Model (KLM, Card et al. 1980) assigns durations: KK = keystroke (0.28s), PP = pointing (1.1s), HH = homing (0.4s), MM = mental operation (1.35s), RR = system response (variable)
  • Methods: sequences of operators and sub-goals that accomplish a goal; each method is a procedural specification for achieving a goal
  • Selection rules: when multiple methods exist for the same goal, rules specify which to use based on contextual conditions

GOMS prediction: total task time = sum of all operator durations executed by the selected methods. This gives quantitative performance predictions without experiments.

GOMS is the first formal cognitive model of task performance: it provides an algorithmic account of how a skilled user performs a task, decomposing it into a structured hierarchy of goals and operators.

Norman: the action cycle

Donald Norman (The Design of Everyday Things, 1988) introduced the action cycle as a model of how intentional action is structured:

  1. Goal formation: forming a desired state
  2. Intention to act: forming an intention toward the goal
  3. Action sequence specification: determining what actions to take
  4. Action execution: executing the actions
  5. Perception of system state: perceiving the resulting state
  6. Interpretation: interpreting the perceived state
  7. Evaluation: evaluating whether the goal has been achieved

Norman’s two gulfs:

  • Gulf of execution: difficulty translating intentions into available actions (gap between intent and affordance)
  • Gulf of evaluation: difficulty interpreting system state (gap between system display and user comprehension)

The action cycle makes explicit that a task is not just an execution sequence but a loop involving perception and evaluation — the task includes the agent’s assessment of whether the goal has been achieved.

Open questions

  • Whether GOMS selection rules (gi⇒mi)(g_i \Rightarrow m_i) for a goal GG are formally equivalent to the guard structure of a procedure — making a GOMS model a procedure specification for cognitive task execution, with each goal-with-selection-rules being a guarded-command procedure.
  • Whether there is a natural notion of task refinement — where T2T_2 refines T1T_1 iff every acceptable execution of T2T_2 is an acceptable execution of T1T_1 (achieving the more specific goal subsumes achieving the more general) — and whether this corresponds to goal subsumption in the HTA hierarchy.
  • Whether the HTA hierarchy corresponds to the tower structure in the relational universe — whether each level of decomposition maps to a tower level, with atomic operations at the base and the overall goal at the top, and whether task correctness (plan-goal coherence) is a property of how these levels connect.
  • Whether the stopping criterion p×cp \times c (probability of failure times cost) has a formal representation as a decision-theoretic threshold on the fiber — whether it corresponds to a condition on the Heyting algebra value of the subtask’s completion proposition, making the p×cp \times c threshold a formal boundedness condition on the fiber.
